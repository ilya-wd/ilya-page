A Look at Web Components

Preface

I‚Äôve used Web Components (or WCs) for the first time to build this (this) website. My use was somewhat rudimentary and primarily concerned with repeated elements, like cards and corresponding dialog elements on the main page. My main goal for my (pretty simple) website was not to use web frameworks and to avoid client-side dependencies.

But I also didn‚Äôt want to write every element in HTML nor use JS with string interpolation as an alternative to templates. Before I started building my website, I had heard about Web Components and decided to try them.

They are pretty neat for a beginner ‚Äì no need to configure any dev tools or install anything.

As they are so neat, a reasonable question arises: is it worth doubling down on them and perhaps using them exclusively, or at least extensively, in my more significant hobby project? To what degree can they replace or augment major web frameworks like React? I decided to read up on it.

Definitions

MDN defines Web Components as:

‚Ä¶a suite of different technologies allowing you to create reusable custom elements ‚Äî with their functionality encapsulated away from the rest of your code ‚Äî and utilize them in your web apps.

As currently many people‚Äôs first touchpoint with the web development is a framework like React, one might quickly conjure up a React component (or Vue or Svelte ‚Äì I‚Äôm only familiar with React, so I will confine myself with the React example).

That association is valid, but only to a limited degree. That is, both have reusability and composability as one of their goals.

But a React component, as Jeremy Keith puts it, relies heavily on its props. Not to mention other parts of the framework like hooks (optional) or renderers like React DOM or diffing (mandatory).

Web Components, on the other hand, are native to the web. They are just Vanilla JS (the largest cross-platform JS framework üòâ), HTML, and CSS. They let you define custom HTML components with logic from JS and styles from CSS and inject them into your code as simply as your ordinary HTML elements.

By native, I mean they have been available in all major browsers since 2020 ‚Äì no need to download any packages to develop or use them.

Web Components are a suite of standards

As you can note from the MDN‚Äôs definition, Web Components is a suite of four web standards.

The four web standards are 4 native web technologies from 4 different HTML specifications.

  1. HTML <template>. JS can consume this template and do something with it if need be

  2. Custom elements. A web developer creates these HTML elements that a browser doesn‚Äôt have (and thus is unaware of) by default. Words in the name of a component are separated by a hyphen, like <my-custom-element>

  3. Shadow DOM. Shadow DOM is a broader topic out of the 4 concepts on this list and deserves a separate post. For now, it suffices to say that it provides encapsulation to elements. It keeps all the logic and styles within the elements, preventing overflow to the original DOM. That shadow barrier, however, can be penetrated ‚Äì scroll to slide 60 and below here to learn more.

  4. ES Modules imports and exports are used to link the components.

Web Components and Modern Web

Let‚Äôs look at the role of Web Components in the web at the beginning of 2024.

Adoption

According to Google, 20% of pages loaded in Chrome are using WC in January of 2024: a twofold increase in 3 years but only 2.5% increase in 2 years.

That‚Äôs a decent number for a technology with a troubled history (more on that below).

It doesn‚Äôt mean that those 20% of loaded pages were built with Web Components only. For example, they could be used in React or some other framework.

Moreover, many popular services and big tech firms, including YouTube, use Web Components. Some of them, like YouTube, have been using them even before Web Components came to browsers via libraries like now-defunct Polymer.

Polymer might be defunct. But as far as Web Components‚Äô ecosystem is concerned, it‚Äôs alive and well (for the most part).

Good things

Here and in the ‚ÄúNot so good things‚Äù section, I will briefly go over rather apparent good and, well, not so good things about Web Components. I will not go into much detail ‚Äì you better check the ‚ÄúFurther reading‚Äù section at the bottom to delve deeper into the topic.

  1. Web Components are platform and framework-agnostic. As a set of W3C standards, they can be used anywhere on the web, in a framework, or without any. You can even embed them in your Markdown files as you do HTML tags. That essentially makes them interoperable ‚Äì and there‚Äôs a handy tool for that.

  2. WCs are native to the web now. That makes them future-proof, and a website built with them will very, very likely still be up and running 20 years from now, just like Space Jam from ‚Äò96.

  3. They are simple. Which can be a double-edged sword, as discussed in the next section.

Not so good things

  1. Implemented with chiefly developers in mind, hence as a tool they are relatively low level.

  2. The above also means that Web Components don‚Äôt have many out-of-the-box functionalities like the ones in modern major frameworks. For example, routing is problematic. As is server side rendering, with some Web Components libraries providing limited SSR to own components. So, you still need some framework to get those handy abstractions in your toolbox.

  3. Problematic marketing and publicity. Started more than a decade ago, Web Components has only begun getting traction in the past few years. Seeing the explosive growth of component-based frameworks on the one hand, many in the dev community questioned the relevance and timeliness of WCs. Now, the tide seems to turn for Web Components, with more and more community members praising WCs in the past year for what they are. The discussion isn‚Äôt one way, though. Here‚Äôs an outstanding article, both in terms of the sentiment and approach to the topic, asserting that Web Components solve problems wrongly and in the wrong way. Definitely worth a read.

  4. They are incompatible with older browsers. The further we go, the less acute this problem becomes, however.

  5. Problems with CSS styling. As Dave Rupert, an advocate for Web Components, puts it:

Styling Web Components is the biggest foot gun of the entire Web Component system because it involves something called ‚ÄúPiercing the Shadow DOM‚Äù. The Shadow DOM is a boundary inside the component where no styles bleed in, and no styles bleed out‚Ä¶ but some styles do pass through!

  1. The fact they are a (set of) Web Standard(s) makes them change really slowly.

Summing up

Comparing the two lists above, one might think the cons outweigh the pros. It‚Äôs not a number competition, however. The simplicity and nativity to the web of WCs make them a good choice for many use cases.

Yes, there‚Äôs a list of problems. Many of them are the sprouts of discussions about the web, as Web Standards are a pretty big thing that reasonably involves a lot of people, many of whom want to pitch in. And yes, these problems are not quick to solve because they reside in Web Standards, which require lots of considerations and thorough planning to change.

On the bright side, there are already established libraries, like Lit or Stencil, that bring better developer and user experiences, and many more smaller ones are being created.

As for my part, I will probably continue to use them in smaller things and will observe how they and the community around them evolve.
